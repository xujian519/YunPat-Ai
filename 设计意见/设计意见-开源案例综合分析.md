## 一、开源社区成功案例的关键启示

在提出具体建议之前，先梳理 2024-2026 年开源/商业 AI 工具领域最值得关注的设计实践，这些实践已经经过大规模用户验证。

### 1.1 Windsurf（Codeium）—「Flow」模式切换的启示

Windsurf 的核心创新不是某个具体功能，而是 **Flows** 概念：AI 在 **Copilot 模式**（辅助型，低延迟，边写边建议）和 **Agent 模式**（自主型，多步执行，跨文件操作）之间**无缝流动**，不需要用户手动切换模式。

**关键数据**：Windsurf 的 Cascade agent 被 OpenAI 以约 $30 亿收购（后转为 Cognition 收购），证明了「Agent-first IDE」方向的产品价值。

**对 YunPat 的启示**：当前文档中 PatentLoop 和 AgentLoop 是**硬分叉**的——用户一旦进入专利标签，系统就判定走 PatentLoop。但专利代理人的实际工作中，大量操作是**辅助型**的（如"帮我查一下这个术语的审查指南定义"），不需要完整的五步流程。如果每次小查询都触发完整的 PatentLoop，会产生不必要的延迟和 token 消耗。

### 1.2 Aider — Git 语义化回滚与 diff 驱动编辑的启示

Aider 是开源社区公认的最佳 AI 结对编程工具，其成功核心在于三个设计决策：

1. **每次 AI 编辑自动 `git commit`**，不是文件快照，而是语义化版本记录
2. **Diff-based 编辑**：AI 输出 SEARCH/REPLACE 块，用户可见精确变更，而非整文件重写
3. **Architect/Editor 双模型模式**：Architect 模型（如 Claude Opus）负责规划，Editor 模型（如 Claude Sonnet）负责执行具体编辑

**关键数据**：Aider 在 GitHub 上有极高的采用率，开发者报告 productivity gain 达 4×，核心原因不是模型更强，而是**工作流设计更符合人类协作直觉**。

**对 YunPat 的启示**：当前文档中的文件回滚使用 TimeMachine 快照（`~/YunPat/workspaces/tab-xxx/.yunpat/snapshots/`），这是文件系统级别的副本。对于法律文书工作，Git 语义化版本管理比文件快照更有价值——可以精确看到"AI 将权利要求 1 的'包括'改为'包含'"的 diff，而不是"恢复到快照 2"的模糊操作。

### 1.3 Claude Code — Plan Mode / Sub-agent / Hooks 的启示

Claude Code 是 Anthropic 的终端原生 agent，2025-2026 年的核心迭代包括：

| 功能 | 设计意图 | 用户价值 |
|------|---------|---------|
| **Plan Mode** (`Shift+Tab` 触发) | 只读探索 → 生成计划 → 用户确认 → 执行 | 避免 AI 在不了解全貌时就动手修改 |
| **Sub-agents** | 主 agent 将任务分解，派生子 agent 并行执行 | 复杂任务分解，降低上下文压力 |
| **Hooks** | 在工具调用前/后执行用户脚本（如 `post-edit: npm run lint`） | 自动化质量门禁，不可被模型覆盖 |
| **Checkpoints** | `/checkpoint save` 语义化书签 | 用户主动标记安全点，非自动快照 |
| **Skills** | `.claude/skills/SKILL.md` 文件系统 | 领域知识和工作流可复用、可共享 |
| **Headless 模式** | `claude -p "prompt" --output-format json` | CI/CD 集成，自动化流水线 |

**对 YunPat 的启示**：PatentLoop 的 Step 3（规划）→ Step 4（执行）可以借鉴 Plan Mode，增加一个**只读分析阶段**；Step 5（检查）可以借鉴 Hooks，将合规检查从 AI 自评升级为用户可配置的**不可覆盖的自动化检查**。

### 1.4 Cursor — Tab 多行预测、Composer 与 .cursor/rules 的启示

Cursor 2026 年的核心实践：

1. **Tab 多行预测编辑**：不是单行补全，而是基于用户近期编辑、打开标签页、代码库索引的**多行预测**。用户按 Tab 接受整个建议，按 Esc 拒绝。接受率比传统补全高 28%，但建议次数少 21%——"说得更少，但更有价值"
2. **`.cursor/rules/*.mdc` 文件系统**：项目级规则文件，自动注入 AI 上下文。团队共享规则，新成员自动获得项目规范
3. **Composer 多文件编辑**：计划 → 跨文件修改 → 统一 diff 审查 → 批量应用/拒绝
4. **并行 Agents**：2026 年 2 月支持最多 8 个并发 agent，使用 git worktrees 隔离
5. **模型路由策略**：Tab 用 Cursor Sonic（自研低延迟模型），Composer 用 Composer-1（自研成本优化模型），Chat 用 Claude Sonnet，困难问题用 Claude Opus——**按任务特征匹配模型，而非统一模型**

**对 YunPat 的启示**：
- 文档工作区的标注系统可以借鉴 Cursor 的 diff 审查模式——AI 的修改以 diff 形式呈现，用户逐项确认
- 专利领域的规则（审查指南要点、撰写规范）可以文件化，类似 `.cursor/rules/`
- 模型路由不应只按用户偏好，应按任务类型自动选择（如检索用长上下文模型，撰写用强推理模型）——文档中有提到，但需强化执行策略

### 1.5 Zed — 显式上下文控制与 Agent 线程的启示

Zed（Rust 原生编辑器，开源 GPL-3）的 AI 设计哲学与 YunPat 文档中的「Deference（退让）」高度一致：

1. **显式上下文控制**：用户用 `@filename` 显式引用文件，而非自动索引。"你知道什么被送入了模型"——这是隐私和可控性的核心
2. **Agent Threads**：多个 agent 线程并行运行，每个线程可以独立配置模型和工具
3. **Terminal Threads**：终端内直接运行 Claude Code / Codex 等外部 agent，Zed 作为渲染和文件同步层
4. **本地模型优先 for inline completion**：Zed 的 Zeta 模型本地运行，预测下一个编辑动作（keystroke 粒度）

**对 YunPat 的启示**：
- 文档工作区的 Agent 感知默认是 `explicit`（用户显式同步），这与 Zed 的显式控制一致。但应考虑增加**半自动模式**（如用户保存文档时自动同步，而非每次编辑都同步）
- 专利标签可以考虑支持多个 Agent 线程——如一个线程在后台检索对比文件，另一个线程在撰写权利要求

### 1.6 MCP 生态 — 工具分组与单职责服务器的启示

MCP（Model Context Protocol）2025 年的最佳实践已经形成：

1. **Tool Grouping Pattern**：工具超过 10-15 个时，按逻辑分组（如 `检索工具组`、`撰写工具组`），LLM 先选组再选工具。这比扁平列表减少认知负荷和错误率
2. **Single Responsibility Server**：每个 MCP server 只做一件事（如一个 server 只接数据库，一个只接文件系统），避免单体 server
3. **Prompt / Tool / Resource 三层分离**：Prompt 是模板，Tool 是动作，Resource 是只读数据。三者不可混用
4. **Tool 元数据丰富化**：除 description 外，还应包含 `cost`、`latency`、`cacheable`、`idempotent`、`sideEffect` 等属性，供 planner 自主调度

**对 YunPat 的启示**：文档中的 ToolRouter 是扁平列表（`listTools()` 返回所有工具）。当专利插件挂载后，工具数量可能达到 50+，需要引入 Capability/Tool Grouping 分层。

## 二、针对 YunPat 文档的具体优化建议

以下建议按**实施阶段**和**重要性**排序，避免与已有两份意见（claude.md、设计建议-chatgpt.md）重复。

### 🔴 第一阶段：必须在 MVP 前解决的设计缺陷

#### 建议 1：引入「Flow 模式」替代硬分叉的 Loop 选择

**问题**：当前 PatentLoop / AgentLoop 的判定是关键词匹配（"含专利检索/撰写/OA/无效/侵权 → PatentLoop"），这是硬分叉。但专利代理人的实际工作流高度混合：在专利案件中写 Python 脚本、查审查指南定义、整理证据材料——不同操作需要不同深度的 AI 参与。

**借鉴**：Windsurf 的 Flows 概念——AI 根据任务复杂度**自动选择**介入深度：
- **Copilot Flow**（辅助）：快速查询、术语解释、格式调整——不需要完整五步，直接回答
- **Guided Flow**（引导）：需要检索或分析——AI 执行 1-2 步，然后暂停确认
- **Agent Flow**（自主）：复杂撰写、OA 答复——完整五步，最小化人机交互

**具体方案**：

```swift
enum AgentFlow {
    case copilot       // 直接响应，不进入 Loop
    case guided        // 进入 PatentLoop，但只走到 Step 2 或 Step 3，暂停确认
    case fullAgent     // 完整五步
}

// 判定逻辑升级：不是关键词匹配，而是轻量 LLM 分类 + 用户可覆盖
// 在 Tab 设置中增加 "默认 Flow 模式" 覆盖选项
```

**价值**：避免小查询触发完整 PatentLoop 的延迟和 token 浪费；用户始终感到 AI "在合适的时候以合适的方式"出现。

---

#### 建议 2：将文件回滚从 TimeMachine 快照升级为 Git 语义化版本管理

**问题**：当前文件回滚使用文件系统快照（`snapshots/` 目录），这是"恢复到某个时间点"的模糊操作。法律文书需要的是"撤销 AI 将权利要求 1 的'包括'改为'包含'"的精确语义化回滚。

**借鉴**：Aider 的 Git 集成——每次 AI 编辑自动 `git commit`，消息描述具体变更。`git reset HEAD~1` 即可精确撤销。

**具体方案**：

```swift
// 替代方案：每个标签/workspace 是一个独立的 Git 仓库（或子模块）
// AI 每次写入文件前，自动执行 git commit

protocol VersionControl {
    func stage(_ changes: [FileChange]) throws
    func commit(message: String, author: CommitAuthor) throws -> CommitID
    func revert(to commit: CommitID) throws
    func diff(from: CommitID, to: CommitID) -> [FileDiff]
}

enum CommitAuthor {
    case user      // 用户手动编辑
    case agent     // AI 自动编辑
    case hybrid    // 人机协作编辑
}

// 提交消息自动生成：
// "[agent] 修改 权利要求书.md: 将'包括'改为'包含'（3处）"
// "[agent] 新增 说明书.md: 技术领域段落"
// "[user]  手动修改 权利要求书.md: 添加从属权利要求 3"
```

**TimeMachine 模式保留**：作为第二层保险，用户可选择"恢复到快照"（粗暴回滚）或"撤销最近 N 个 agent 提交"（精确回滚）。

**价值**：
- 法律文书可审计——每个修改都有"谁、何时、改了什么"的精确记录
- 回滚粒度精确——撤销一次 AI 操作，而非恢复到 10 分钟前的全部状态
- 与外部协作兼容——可将 workspace 直接推送到 Git 仓库与团队共享

---

#### 建议 3：PatentLoop Step 3（规划）增加「Plan Mode」— 只读探索阶段

**问题**：当前 Step 3 直接生成 plan 然后进入 Step 4 执行。但对于复杂案件（如无效宣告），AI 在不了解全部技术事实和对比文件的情况下就生成 plan，可能方向错误，导致后续大量返工。

**借鉴**：Claude Code 的 Plan Mode（`Shift+Tab` 触发）——先只读探索代码库，生成计划，用户确认后再执行。

**具体方案**：

```swift
enum PlanMode {
    case auto        // 当前行为：直接生成 plan → 执行
    case interactive // 新增：先只读探索 → 生成 plan → 用户确认/修改 → 执行
    case readOnly    // 新增：只读探索 → 生成 plan → 不执行，仅呈现给用户
}

// Step 3 内部流程升级：
step3_buildPlan(facts, rules):
    if planMode == .interactive:
        // 1. 只读探索：读取相关 wiki 页面、检索对比文件、分析现有权利要求
        let exploration = await readOnlyExploration(facts, rules)
        // 2. 生成 plan 草案
        let planDraft = await generatePlan(exploration)
        // 3. 协作面板呈现：策略树 + 每一步的规则依据 + 预期产出
        // 4. 用户可：确认 / 修改策略树 / 要求重新探索
        let approvedPlan = await userApproval(planDraft)
        return approvedPlan
    else:
        return await generatePlan(facts, rules) // 当前行为
```

**价值**：对于复杂案件，避免 AI 在不了解全貌时就动手；plan 的可解释性提升用户信任。

### 🟡 第二阶段：架构增强（MVP 后尽快实现）

#### 建议 4：引入专利领域「Rules 文件系统」— 类似 `.cursor/rules/`

**问题**：当前知识库集成是通过 WikiAdapter 读取已有的宝宸知识库。但专利代理人在**不同案件类型**中需要的规则是不同的——化学案件需要关注实验数据，机械案件需要关注结构连接，软件案件需要关注功能性限定。这些规则应该在**项目/案件级别**可配置。

**借鉴**：Cursor 的 `.cursor/rules/*.mdc` 文件系统——项目根目录下的规则文件自动注入 AI 上下文。团队共享，新成员自动获得规范。

**具体方案**：

```
~/YunPat/workspaces/tab-001/
├── .yunpat/
│   ├── rules/
│   │   ├── 01-案件类型-化学.md        # 化学领域特殊规则
│   │   ├── 02-撰写规范.md             # 事务所统一撰写规范
│   │   ├── 03-客户偏好-华为.md        # 特定客户偏好
│   │   └── 04-当前策略-发明授权.md    # 当前案件策略约束
│   ├── snapshots/
│   └── history.json
├── input/
└── output/
```

```swift
// 启动时自动读取 .yunpat/rules/*.md，按文件名排序注入系统 prompt
// 规则文件优先级：案件级 > 客户级 > 事务所级 > 默认

struct RuleFile {
    let path: URL
    let priority: Int     // 文件名前缀数字
    let scope: RuleScope // .case / .client / .firm / .default
    let content: String
}

// 注入策略：
// 1. 案件级规则（当前 workspace/.yunpat/rules/）
// 2. 客户级规则（~/.yunpat/templates/rules/{client}/）
// 3. 事务所级规则（~/.yunpat/templates/rules/default/）
// 4. 知识库（宝宸知识库 WikiAdapter）
```

**价值**：
- 不同案件类型自动加载不同规则——化学案件自动加载"实验数据支持"规则，软件案件自动加载"功能性限定"规则
- 团队协作——资深代理人写好规则文件， junior 自动获得指导
- 客户定制——特定客户（如华为）的特殊要求自动注入

#### 建议 5：MCP 工具引入「Capability / Tool Grouping」分层

**问题**：当前 ToolRouter 是扁平列表。已有意见（设计建议-chatgpt.md）提出 Capability Layer 概念，本建议补充 MCP 生态验证的实现细节。

**借鉴**：MCP 2025 最佳实践中的 Tool Grouping Pattern——当工具超过 10-15 个时，按逻辑分组。

**具体方案**：

```swift
// 在 ToolDefinition 之上增加 Capability 层

struct CapabilityDefinition: Codable, Sendable {
    let name: String           // "patent.search"
    let displayName: String    // "专利检索"
    let description: String    // LLM 用于理解能力边界
    let tools: [ToolDefinition]  // 该能力下的工具
    let source: CapabilitySource
    let permission: CapabilityPermission
    let metadata: CapabilityMetadata
}

struct CapabilityMetadata {
    let estimatedLatency: TimeInterval  // 预估延迟（供 planner 使用）
    let costLevel: CostLevel            // .free / .low / .medium / .high
    let requiresNetwork: Bool
    let isIdempotent: Bool              // 是否幂等（可重试）
    let typicalUseCases: [String]       // 典型使用场景
}

// LLM 工具选择流程升级：
// 1. 从所有 Capability 中选择匹配的（5-10 个）
// 2. 从选中 Capability 的工具中选择具体工具（每个 Capability 3-5 个）
// 总决策空间：25-50 个，而非 100+ 个扁平工具
```

**专利场景示例**：

```
Capability: "patent.search"
  ├── tool: search_cnipa        // 国知局检索
  ├── tool: search_google       // Google Patents
  ├── tool: search_wipo         // WIPO Patentscope
  └── tool: search_local_db     // 本地对比文件库

Capability: "patent.drafting"
  ├── tool: draft_claims        // 权利要求起草
  ├── tool: draft_description   // 说明书起草
  └── tool: check_format        // 格式检查

Capability: "patent.oa"
  ├── tool: analyze_oa          // OA 分析
  ├── tool: compare_features    // 特征对比
  └── tool: draft_response      // 答复稿生成
```

**价值**：减少 LLM 工具选择时的"选择困难"和错误率；当插件工具扩展到 50+ 时，系统仍然可控。

#### 建议 6：引入「Architect / Editor 双模型」模式用于复杂撰写

**问题**：当前 PatentLoop Step 4（执行）内嵌 AgentLoop，使用同一模型完成规划和执行。但对于复杂权利要求布局或 OA 答复策略，规划需要强推理（如 Claude Opus），执行需要高效率（如 Claude Sonnet）。单一模型无法兼顾。

**借鉴**：Aider 的 Architect 模式——Architect 模型（强推理）生成计划，Editor 模型（高效率）执行具体编辑。

**具体方案**：

```swift
enum ExecutionMode {
    case singleModel          // 当前行为：同一模型完成全部
    case architectEditor      // 新增：Architect 规划 → Editor 执行
}

// 使用场景：
// - 简单任务（术语查询、格式调整）：singleModel
// - 复杂任务（权利要求布局、OA 策略）：architectEditor

// Architect 模型职责：
// 1. 分析技术事实和规则
// 2. 生成权利要求树结构 / 答复策略树
// 3. 标记每个步骤需要的规则依据
// 4. 输出：结构化 Plan（非自然语言）

// Editor 模型职责：
// 1. 按 Architect 的 Plan 执行具体撰写
// 2. 每步完成后自我验证（如检查引用关系）
// 3. 输出：具体文本内容
```

**模型路由策略**：

| 任务类型 | 模式 | Architect 模型 | Editor 模型 |
|---------|------|---------------|------------|
| 术语查询 | singleModel | Claude Sonnet | — |
| 权利要求撰写 | architectEditor | Claude Opus | Claude Sonnet |
| OA 答复 | architectEditor | Claude Opus | Claude Sonnet |
| 简单检索 | singleModel | Claude Sonnet | — |
| 翻译 | singleModel | GPT-4o / DeepSeek | — |

**价值**：复杂任务的质量由强推理模型保障，成本由高效率模型控制。对于专利撰写这种"规划错误代价极高"的场景，双模型模式的 ROI 明确。

---

#### 建议 7：Step 5（检查）升级为「Hooks + AI 自评」双层检查

**问题**：当前 Step 5 是 PatentReviewer（AI 自评）。但 AI 的自评能力有限——它可能遗漏自己犯的错误。Claude Code 的 Hooks 设计证明：不可覆盖的自动化检查是质量保障的关键。

**借鉴**：Claude Code 的 Hooks 系统——在工具调用后自动执行用户脚本，检查安全/质量规则，不可被模型覆盖。

**具体方案**：

```swift
// 检查层双层设计：

// 第一层：AI 自评（当前已有）
// PatentReviewer 检查：合规性 / 完整性 / 一致性 / 引用检查

// 第二层：用户可配置的 Hooks（新增）
protocol CheckHook {
    var name: String { get }
    var trigger: HookTrigger { get }
    var action: HookAction { get }
    func run(_ context: CheckContext) async throws -> HookResult
}

enum HookTrigger {
    case afterClaimDraft       // 权利要求起草完成后
    case afterOAResponse      // OA 答复完成后
    case beforeFileExport      // 文件导出前
    case onDocumentChange     // 文档变更时
}

enum HookAction {
    case warn(message: String)    // 警告，不阻止
    case block(message: String)   // 阻止，必须修正
}

// 示例 Hooks：
// 1. "权利要求项数检查"：超过 10 项时警告（费用考量）
// 2. "引用关系检查"：从属权利要求引用的独立权利要求是否存在
// 3. "术语一致性检查"：全文中同一技术术语的表述是否一致
// 4. "客户敏感词检查"：是否包含客户禁用词汇
// 5. "法条引用格式检查"：法条引用是否符合事务所规范
```

**Hook 脚本引擎**：
- 支持 Swift 脚本（通过 Swift Scripting）
- 支持 JavaScript/TypeScript（通过 JavaScriptCore）
- 支持正则表达式规则（最简单场景）

**价值**：质量检查从"AI 说没问题"升级为"用户定义的规则 + AI 自评"双层保障。对于专利这种对错误零容忍的领域，这非常关键。

### 🟢 第三阶段：体验优化与长期演进

#### 建议 8：文档工作区增加「Diff 审查模式」

**问题**：当前文档工作区的标注系统（`~~删除~~`、`<ins>插入</ins>`、`**???**`）是用户手动标注，AI 被动感知。但缺少 AI 主动修改后的**审查界面**——用户如何逐项确认 AI 的修改？

**借鉴**：Cursor 的 Composer diff 视图——AI 修改后，用户看到每个文件的 diff，可以逐项接受/拒绝。

**具体方案**：

```swift
// 当 AI 修改文档后，自动进入 Diff 审查模式：

struct DiffReviewSession {
    let originalDocument: WorkspaceDocument
    let proposedDocument: WorkspaceDocument
    let changes: [DocumentChange]  // 结构化变更列表
    let status: ReviewStatus       // .pending / .partiallyAccepted / .fullyAccepted / .rejected
}

// UI 呈现：
// ┌────────────────────────────────────────────┐
// │ Diff 审查: 权利要求书.md                      │
// │                                              │
// │ 变更 1/3:                                    │
// │   - 包括以下步骤：                            │
// │   + 包含以下步骤：                            │
// │   [接受] [拒绝] [修改]                       │
// │                                              │
// │ 变更 2/3:                                    │
// │   - 所述装置包括 ...                         │
// │   + 所述装置包含 ...                         │
// │   [接受] [拒绝] [修改]                       │
// │                                              │
// │ [接受全部] [拒绝全部] [批量处理]             │
// └────────────────────────────────────────────┘
```

**与标注系统的关系**：
- Diff 审查模式：AI 修改 → 用户逐项确认（**结构化**）
- 标注系统：用户手动标注 → AI 感知（**非结构化**）
- 两者互补：AI 主动修改走 Diff 审查，用户即兴想法走标注

**价值**：避免 AI "悄悄"修改文档用户却不知道改了什么；建立用户对 AI 的信任。

#### 建议 9：引入「Agent 线程」概念支持多任务并行

**问题**：当前设计是一个标签一个 Agent Loop。但专利代理人的实际工作高度并行——一个案件可能需要同时：后台检索对比文件、撰写权利要求、分析 OA 意见。如果单线程串行，效率极低。

**借鉴**：Zed 的 Agent Threads——多个 agent 线程并行运行，每个线程可以独立配置模型和工具。

**具体方案**：

```swift
// 在 Tab 内支持多个 Agent Thread：

struct AgentThread {
    let id: UUID
    let name: String              // "对比文件检索" / "权利要求撰写" / "OA 分析"
    let modelConfig: ModelConfig  // 每个线程可配置不同模型
    let toolFilter: ToolFilter    // 每个线程可限制可用工具
    let status: ThreadStatus
    let loopState: LoopState
}

// Tab 结构升级：
struct Tab {
    let id: UUID
    var title: String
    var type: TabType
    var workspace: WorkspaceConfig
    var memory: SessionMemory
    var messages: [ChatMessage]           // 主线程对话
    var agentThreads: [AgentThread]       // 新增：并行线程
    var activeDocument: WorkspaceDocument?
}

// 使用场景：
// 主线程：用户与 AI 对话，协调各子线程
// 子线程 1：后台检索对比文件（使用 Search Capability，低优先级模型）
// 子线程 2：撰写权利要求（使用 Draft Capability，高优先级模型）
// 子线程 3：格式检查（使用 Check Hooks，快速模型）
```

**线程间通信**：

```swift
// 子线程完成后，结果注入主线程的 Session Memory
// 用户可在主线程中引用子线程结果："把子线程 A 检索到的对比文件用于权利要求修改"
```

**价值**：代理人可以在等待检索结果的同时进行撰写；复杂案件的多任务并行处理效率显著提升。

#### 建议 10：知识库集成增加「规则文件 → 知识库」的双向同步

**问题**：当前知识库集成是"知识库 → Agent"的单向读取。但专利代理人在案件中积累的经验（如某类技术领域的创造性判断策略）应该**回流到知识库**。

**借鉴**：Karpathy Wiki 的「Query→Archive 归档流程」——高质量问答沉淀为 wiki 页面。但当前设计是用户手动归档，缺少自动化机制。

**具体方案**：

```swift
// 当案件结案时，自动分析 Case Context 中的高质量产出：
// 1. 识别"新策略"——与知识库中已有策略不同的处理方法
// 2. 生成知识卡片草稿（标注 author: yunpat-ai）
// 3. 用户确认后写入知识库 cards/ 目录
// 4. 更新 Concept-Index.md

// 另外，支持规则文件 → 知识库的同步：
// 当用户在 .yunpat/rules/ 中新增规则文件时，提示是否写入知识库作为通用规范

protocol KnowledgeSync {
    func detectNovelStrategies(from caseContext: CaseContext) -> [StrategyCard]
    func proposeArchive(_ cards: [StrategyCard]) -> ArchiveProposal
    func syncRulesToKnowledgeBase(_ rules: [RuleFile]) -> SyncResult
}
```

**价值**：知识库不是静态的，而是随案件处理不断进化；事务所的集体智慧不断积累。

#### 建议 11：oMLX 本地推理的 fallback 策略需要具体化

**问题**：当前文档中 oMLX 是"初期占位"，但 2026 年的本地模型生态（Ollama、vLLM、llama.cpp）已经非常成熟。对于专利代理人的隐私需求（法律文书不上云），本地推理不是可选项，而是**必须**。

**借鉴**：Continue.dev 的本地模型策略——Ollama 用于个人开发，vLLM 用于团队共享推理服务器。

**具体方案**：

```swift
// oMLX 后端增加 fallback 策略：

enum LocalModelStrategy {
    case ollama          // 个人：Ollama 本地运行
    case vLLM            // 团队：共享推理服务器
    case mlxDirect       // Apple MLX 直接推理
}

// 网络不可用时的自动降级：
// 1. 检测网络状态
// 2. 网络可用 → 使用云端 API（OpenAI/Claude/DeepSeek）
// 3. 网络不可用 → 自动切换到本地模型（oMLX/Ollama）
// 4. 本地模型不可用 → 提示用户并进入离线模式（仅使用知识库文件直读，无 LLM）

// 本地模型用途分配：
// - 简单查询（术语解释、格式检查）：本地 7B-13B 模型
// - 检索增强（语义搜索）：本地 Embedding 模型（bge-m3-mlx）
// - 复杂推理（权利要求布局）：云端模型（Claude Opus）
```

**价值**：对于处理敏感客户信息的专利代理人，"法律文书不上云"是硬性需求。本地推理的 fallback 策略是产品可用性的底线。

#### 建议 12：UI 性能设计需要提前规划

**问题**：文档中 UI 设计非常详细（Liquid Glass、Deference、排版等），但缺少性能考量。当 AI 同时修改多个文档、协作面板实时更新、内嵌浏览器渲染 PDF 时，SwiftUI 的性能可能成为瓶颈。

**借鉴**：Zed 的 Rust + GPU 渲染架构——60fps 始终，AI 修改 50 个文件时"你看到每个变化实时发生"。Electron 架构（VS Code）在 AI 大规模编辑时会出现卡顿。

**具体方案**：

```swift
// 文档工作区性能优化：
// 1. 文本编辑使用 NSTextView（AppKit）而非 SwiftUI TextEditor
//    SwiftUI TextEditor 在大文件（>1MB 的说明书）时性能差
// 2. Diff 渲染使用 Metal 加速（如 JXBrowser 或自研 Diff 渲染器）
// 3. 文件监听使用 FSEvents 的批量模式，而非每个文件单独事件
// 4. 协作面板更新使用 Coalescing——多个事件合并为一次 UI 更新

// 内存管理：
// - 每个标签的 Session Memory 在后台时压缩（如 tieredCompact）
// - 文档工作区只加载当前可见文档，其他文档仅保留索引
// - 知识库全文缓存 LRU 策略（最大内存阈值自动淘汰）
```

**价值**：专利文件动辄数万字，权利要求书可能包含几十项。如果 AI 同时修改多个文件时 UI 卡顿，用户体验会急剧下降。

## 四、总结

YunPat-Ai 的设计文档已经展现了非常罕见的深度——PatentLoop 五步流程、双消费者知识库模式、三层记忆体系、Liquid Glass UI 设计，这些都是经过深思熟虑的架构决策。

但从开源社区的成功案例来看，**2025-2026 年的 AI 工具竞争已经从"功能竞赛"转向"工作流竞赛"**。用户选择的不是"哪个工具功能更多"，而是"哪个工具更懂我的工作方式"。

以下是基于案例分析的**三个核心原则**，建议贯穿整个产品演进：

### 原则 1：AI 应该在合适的时候以合适的方式出现（Flows）

不是每次交互都需要完整的 Agent Loop。 Windsurf 的 Flows 和 Zed 的显式控制共同证明：用户需要**可预测的 AI 介入深度**。YunPat 的 Flow 模式设计（Copilot → Guided → Full Agent）是实现这一点的关键。

### 原则 2：法律文书需要 Git 级别的可审计性，不是快照级别的回滚

Aider 的成功证明，开发者最信任的是 Git 语义化历史。对于专利文件，"恢复到 10 分钟前的快照"不如"撤销 AI 将权利要求 1 的'包括'改为'包含'"精确。Git 语义化回滚是法律场景的**刚需**。

### 原则 3：质量检查必须是「不可覆盖的」+「AI 自评」双层

Claude Code 的 Hooks 是**不可被模型覆盖**的自动化检查，这与 PatentReviewer 的 AI 自评是互补的。对于专利这种"一个错误可能毁掉整个申请"的场景，双层检查不是过度设计，而是底线。
